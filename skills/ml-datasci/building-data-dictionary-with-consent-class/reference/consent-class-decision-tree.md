# Consent-Class Decision Tree

Walk this tree per field to land on one of: `collected`, `inferred`, `derived`, `public`, `synthetic`.

```
Did the user directly provide this value (form, file upload, OAuth scope, API write)?
├─ Yes → collected
└─ No → next question

Is the value the output of a machine-learning model or rule predicting a label about the user?
├─ Yes → inferred
│        (Examples: predicted churn, predicted gender from name, predicted sentiment, risk score,
│         lead-quality label, automated intent classification)
└─ No → next question

Is the value a deterministic transformation / aggregation of fields about this user?
├─ Yes → derived
│        (Examples: lifetime value = sum of transactions, full_name = first + " " + last,
│         user_embedding = encoder(user_text), age = today - dob, hash(email))
└─ No → next question

Was the value obtained from a documented public source with a recorded URL and access timestamp?
├─ Yes → public
│        (Examples: Companies House officer record, public GitHub profile snapshot,
│         government open-data export, scraped news article with cite)
│        REQUIRE: source URL + access timestamp recorded in the dictionary row.
│        Without that, downgrade to "collected" (you might have obtained it lawfully,
│        but it is not provably public-domain at audit time).
└─ No → next question

Was the value generated by a process provably independent of any real-user data
(Faker, public-source-only SDG, deterministic test fixtures)?
├─ Yes → synthetic
│        (Examples: Faker name + address for QA fixtures; SDG output trained ONLY on
│         a public benchmark; deterministic seeded test data)
│        GUARD: if the SDG was fit on real user data, classify as the upstream class
│        (collected / inferred / derived) — synth from real is NOT consent-class synthetic.
│        Hand off to auditing-synthetic-data-leakage to verify the privacy claim.
└─ No → escalate; the field has unclear provenance. Refuse to classify silently;
        flag for source-of-truth investigation before proceeding.
```

## Common reclassification traps

### Trap 1: "It's public because the user told other people too"

A field where the value happens to also be public in some other context (e.g. user's email also appears on their public website) is NOT consent-class `public`. The instance of the data your system holds was collected from the user. Classify as `collected`. The `public` class is reserved for fields obtained directly from a documented public source.

### Trap 2: "It's inferred, so the user has no rights to it"

Inferred-about-a-user is still about the user. GDPR Article 15 (right of access) explicitly covers profiling outputs (Recital 63). GDPR Article 22 covers automated decision-making based on inferred fields. Classify as `inferred` and place in DSR scope.

### Trap 3: "It's derived, so deleting the upstream is enough"

Derived fields propagate. Deleting the upstream does not auto-delete cached embeddings, aggregations baked into a model checkpoint, materialized views, or downstream copies in analytics warehouses. Classify as `derived` AND list every downstream copy in the `propagated_to` column.

### Trap 4: "Synthetic data has no consent class"

Synth-from-real-user-data inherits the upstream consent class until a privacy audit (DCR / NNDR / MIA) proves no membership signal remains. Until that audit, treat synth-from-real as the upstream class for DSR scope. Only Faker-style or public-source-only SDG output is consent-class `synthetic`.

### Trap 5: "Hashed = anonymous"

A hash of an email is still pseudo-identifying — the same user always hashes the same value, so it joins across systems. Classify as `derived` from `collected` upstream; remains in DSR scope until provably k-anonymized. The hash is not magic; it is a deterministic transformation.

### Trap 6: "It was inferred from a public field, so it's public"

Inferring a label about a user from a public field (e.g. predicting political affiliation from a public donation record) still creates an inferred attribute about a specific person. Classify as `inferred`, not `public`. The upstream being public is irrelevant to the consent class of the output about a person.
