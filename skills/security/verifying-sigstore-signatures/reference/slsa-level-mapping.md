# SLSA Build Level mapping checklist

Per SLSA v1.0 (Build track). L4 was retired in v1.0 — the historical L4 criteria are now distributed across L3 plus a separate Source track. Do not claim L4.

## Build L0

- No provenance, or provenance unverifiable.
- Verdict path: no attestation OR Check 3 fails entirely.

## Build L1

**Criteria** (all required):

- [ ] Provenance exists as a verifiable artifact (in-toto envelope or equivalent)
- [ ] Predicate contains `subject` (the artifact identifier) and `buildType`
- [ ] Predicate is consumable by downstream verification tooling

**Note**: provenance signed by the *producer themselves* (self-attestation) maps to L1 at best. Self-attestation provides auditability but not tampering resistance — the producer can fabricate the predicate at any time.

**Predicate-field test**:
- `builder.id` may be the producer's identity
- `predicate.invocation.configSource` may be absent

## Build L2

**Criteria** (all L1 criteria + the following):

- [ ] Provenance is signed by a *hosted build platform* OIDC identity, not by the producer's own identity
- [ ] The build platform generates the predicate content (not the producer's build script)
- [ ] Predicate references the source: `predicate.invocation.configSource.uri` AND `predicate.invocation.configSource.digest` are present and resolvable
- [ ] Signature is verifiable via Sigstore (Rekor entry present) OR equivalent transparency log

**Predicate-field test**:
- `builder.id` is an OIDC subject from a recognized build platform (e.g., `https://github.com/actions/runner/.../slsa-generic-generator.yml@refs/tags/v1.x.x`)
- `predicate.invocation.configSource.uri` is a resolvable URL (e.g., `git+https://github.com/example/repo@refs/heads/main`)
- `predicate.invocation.configSource.digest` is the source-revision SHA the build was launched from

## Build L3

**Criteria** (all L2 criteria + the following):

- [ ] Build platform isolates the build environment such that the build process cannot inject false provenance content at runtime (the predicate is generated by trusted-control-plane code, not by user-controlled build scripts)
- [ ] Build platform's isolation guarantees are documented and publicly verifiable
- [ ] Provenance generation runs in a separate trust boundary from user-controlled build steps

**Predicate-field test**:
- `builder.id` is a SLSA-conformant builder (currently includes: GitHub Actions SLSA reusable workflows in `slsa-framework/slsa-github-generator`, GoReleaser w/ Sigstore configured for SLSA L3, Tekton Chains with SLSA L3 configuration)
- The builder's documentation explicitly claims L3 conformance and the predicate's `builder.id` resolves to that documented builder
- The producer's own CI steps cannot mutate the generated provenance

**Common L3 false positives**:

- A custom GitHub Actions workflow that *uses* OIDC tokens but *generates* its own provenance is L2 at best — the user-controlled workflow can fabricate the predicate
- A self-hosted GitHub runner controlled by the producer breaks the isolation requirement — L2 at best
- Cosign sign with `--predicate` from a user script generates a Producer-controlled predicate — L1 at best

## Build L4 — DO NOT USE

L4 was retired in SLSA v1.0. Operators claiming L4 are referencing the pre-v1.0 spec. Re-map their claim to L3 with the relevant Source-track or Deployment-track criteria separately, or treat the claim as L3 unless the operator can name a current SLSA spec section that supports L4.

## Quick verdict table

| Producer-controlled signing? | Hosted-platform signing? | Isolated build platform? | SLSA Build level |
|---|---|---|---|
| Yes | — | — | L1 |
| No | Yes | No | L2 |
| No | Yes | Yes (documented + verifiable) | L3 |
